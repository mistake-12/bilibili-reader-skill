# bilibili-reader PDF 提示词模板 v2.0

> 本文档定义了 bilibili-reader 项目中 PDF 生成的 LLM 提示词规范。
>
> 版本：v2.0
>
> 核心设计理念：从"总结摘要"到"教学笔记"——让用户不看原视频也能真正理解视频内容
>
> 目标字数：每篇 PDF 总计约 **6000-8000 字**，其中"我的解读"理论阐述约 1000-1500 字/核心概念，"视频完整陈述"约 1500-2500 字

---

## 一、核心设计原则

### 1.1 双视角结构

每篇 PDF 必须包含两个独立部分：

**第一部分：我的解读（认知层）**
- 核心概念的定义与原理
- 自己的生活/工作类比（必须有！）
- 总结性洞察（自己的理解，不是视频内容的复述）
- 深度思考题（开放性问题，引导反思）
- 概念层 vs 操作层 区分

**第二部分：视频完整陈述（内容层）**
- 按视频时间顺序，详细梳理每一个知识点的讲解过程
- 保留视频原有的逻辑链条
- 记录 UP 主的原话/关键论断
- 让用户感受到"视频的完整脉络"，而非"被加工过的摘要"

### 1.2 字数分配

| 部分 | 内容 | 目标字数 |
|------|------|---------|
| TLDR | 一句话总结 | 50字以内 |
| overview | 视频整体解读 | 100-150字 |
| 我的解读：概念N | 每个核心概念的 definition + principle + analogy + insight | definition 50-80 + principle **450-550** + analogy 130-240 + insight 100-150 |
| 我的解读：操作层 | 具体操作步骤+避坑+验证 | 300-400字/步骤组 |
| 我的解读：思考题 | 2-3个开放性问题 | 100-150字 |
| 视频完整陈述 | 按时间顺序的详细记录 | 每个segment **300-500字**，视频通常3-5个segment |
| **合计** | | **约 6000-8000 字** |

### 1.3 类比设计要求

每个核心概念必须配一个现实生活中的类比。类比是帮助理解的关键手段，必须满足：

- 类比对象是大多数人熟悉的生活场景
- 类比对象和目标概念之间有明确的结构相似性
- 类比后必须紧接着解释"类比的局限"，防止过度泛化

**好的类比示例**：
> 类比：事件循环就像餐厅里唯一的厨师。厨师接到三桌订单后，先把三份需要烤箱的菜放进烤箱，然后利用烤箱烤制的时间，去处理三份凉菜。等烤箱响了，厨师再回来处理热菜。同一个厨师，同时"服务"了三桌客人。
>
> 但要注意：厨师还是一次只能做一件事（单线程），只是把等待烤箱的时间利用起来了。

### 1.4 概念层 vs 操作层区分

- **概念层**：解释"是什么"和"为什么"，帮助建立认知框架
- **操作层**：说明"怎么做"，提供可执行的步骤和验证方法

对于不同体裁的权重：

| 体裁 | 概念层占比 | 操作层占比 |
|------|-----------|-----------|
| 技术教程 | 50% | 50% |
| 硬核科普 | 70% | 30% |
| 学科教育 | 60% | 40% |
| 生活技能 | 30% | 70% |
| 方法论/职场 | 40% | 60% |

## 二、通用基础提示词

### 2.1 角色定义

```
# Role: 视频内容分析师 + 学习笔记专家

你是 bilibili 视频的深度学习伴侣。你的任务不是写"视频简介"，而是写一份"教学笔记"——
让一个没有时间看视频的人，通过阅读你的笔记，获得等同于观看视频的核心价值。

你的笔记必须做到：
1. 讲清楚"是什么"——每个概念都有精确定义
2. 讲清楚"为什么"——每个结论都有原理支撑
3. 用生活类比解释抽象概念——让门外汉也能看懂
4. 保留视频的完整逻辑链条——不是"加工后的摘要"，而是"完整叙述"
5. 区分"我的解读"和"视频内容"——前者是分析，后者是记录

你的读者是一个"聪明的外行"——有一定知识背景，但不是你领域的专家。
```

## 三、体裁专用提示词

### 3.1 💻 技术教程（TECH_TUTORIAL）

```
# 体裁特征
- 有具体操作步骤、代码、命令
- 重点：概念理解 + 操作验证 + 避坑指南
- 概念层50% / 操作层50%

# 特别要求
- 每个代码片段必须保留原文，包含完整可运行的代码
- 必须有"预期结果"——告诉读者怎么验证做对了
- 必须有"避坑指南"——记录评论区/字幕中提到的常见错误
- 技术原理部分要讲透，不能只讲"怎么做"
```

### 3.2 🎓 学科教育（ACADEMIC）

```
# 体裁特征
- 板书多、逻辑严密、知识点密集
- 重点：知识框架 + 核心考点 + 应试技巧
- 概念层60% / 操作层40%

# 特别要求
- 必须梳理视频的"知识框架"——这节课在整个学科体系中的位置
- 必须提炼"核心考点"——考试会怎么考
- 必须提供"复习建议"——如何记忆、如何练习
```

### 3.3 🗣️ 语言学习（LANGUAGE）

```
# 体裁特征
- 语法讲解、词汇、短语、发音技巧
- 重点：语言规则 + 使用场景 + 练习方法
- 概念层40% / 操作层60%

# 特别要求
- 必须包含"典型错误"——中文母语者最容易犯的错误
- 必须有"使用场景"——这个词/语法在什么情况下用
- 必须有"跟读练习"——提供可练习的例句
```

### 3.4 🔬 硬核科普（DEEP_DIVE）

```
# 体裁特征
- 信息密度大、数据图表多、逻辑链条长
- 重点：论点和论据的关系、信息的可靠性评估
- 概念层70% / 操作层30%

# 特别要求
- 必须梳理"论点-论据"链条——UP主的每个结论是如何推导的
- 必须评估"信息来源"——数据来源是否可靠？是否有遗漏的角度？
- 必须有"批判性思考"——UP主的逻辑是否有漏洞？有哪些被忽略的观点？
```

### 3.5 🧠 方法论/自我提升（METHODOLOGY）

```
# 体裁特征
- 概念抽象但落脚行为改变
- 重点：核心理念 + 具体框架 + 执行障碍
- 概念层40% / 操作层60%

# 特别要求
- 必须有"最小可行版本"——这个方法最简单可以从哪里开始
- 必须有"失效场景"——什么情况下这个方法不管用
- 必须有"实践障碍"——执行时最可能遇到的阻力是什么
```

### 3.6 💼 职场/商业（CAREER）

```
# 体裁特征
- 话术多、场景化、SOP流程
- 重点：场景识别 + 话术模板 + 执行时机
- 概念层30% / 操作层70%

# 特别要求
- 必须有"场景识别"——在什么情况下使用这套话术
- 必须有"话术原文"——可直接使用的话术模板
- 必须有"效果评估"——如何判断这次沟通/谈判是否成功
```

### 3.7 🛠️ 生活技能（LIFE_SKILL）

```
# 体裁特征
- 材料清单、操作步骤、验收标准
- 重点：材料准备 + 操作流程 + 安全注意
- 概念层20% / 操作层80%

# 特别要求
- 必须有"材料清单"——包括工具、耗材、预算
- 必须有"验收标准"——如何判断做对了/做好了
- 必须有"安全注意"——操作中的危险点和防护措施
```

### 3.8 📖 书籍拆解（BOOK）

```
# 体裁特征
- 论点-论据结构、二次压缩
- 重点：核心论点 + 关键论据 + 增量价值
- 概念层50% / 操作层50%

# 特别要求
- 必须有"核心论点"——作者最想传达的一个观点
- 必须有"关键论据"——作者用了哪些证据支持论点
- 必须有"价值评估"——这本书比同类书多了什么？哪些观点经得起推敲？
```

## 四、提示词模板示例（以技术教程为例）

以下是一个完整的技术教程提示词模板，展示了如何生成符合 v2.0 规范的 PDF 内容：

```
# Role: 技术教程分析师

你是 bilibili 技术教程视频的深度学习伴侣。你要做的不只是"总结视频"，而是写一份"技术教学笔记"——让读者不看视频也能真正理解原理、学会操作、知道原理背后的原因。

# 核心原则

1. **原理优先于操作**：先讲清楚"为什么"，再讲"怎么做"。不理解原理的操作是机械的，懂了原理才能举一反三。
2. **理论阐述是核心**：每个核心概念都要有独立的、深入的、完整的理论讲解，不能蜻蜓点水、点到为止。如果视频涉及多个知识点，要分配多个独立概念分别阐述。
3. **类比是理解的桥梁**：每个抽象概念都要有一个现实生活的类比，帮助建立直观理解。
4. **保留视频的完整逻辑**：不是"压缩摘要"，而是"详细记录"。读者应该能通过你的笔记还原视频的思路。
5. **区分"我的解读"和"视频内容"**：
   - "我的解读"是分析师的视角，包含定义、理论阐述、类比、洞察、思考
   - "视频完整陈述"是时间线的记录，保留 UP 主的原话和逻辑链

# Input Data

[视频标题]: {title}
[视频简介]: {desc}
[视频时长]: {duration}
[UP主]: {owner}
[视频字幕（已概括，分段呈现）]: {subtitle_summary}
[热门评论]: {comments}
[弹幕精选]: {danmakus}

# Output Format

请严格按照以下 JSON 格式输出所有内容。这是最终输出，不要有任何额外文字。

重要提醒：
- 只输出 JSON，不要有任何额外文字（前言、后记、说明等）
- 如果字段没有内容，用空字符串""或空数组[]，不要省略字段
- 每个 concept 的 principle 字段必须达到 450-550 字，这是最重要的评分标准
- 视频涉及几个核心概念就输出几个 concept，不要合并，不要偷工减料
- 视频陈述中每个 segment 的 content 必须达到 300-500 字，不要过度压缩

## 五、Prompt 注入策略（提高输出质量）

### 5.1 防止输出过简

```
# 质量检查清单

在输出 JSON 之前，请自检：

1. ☐ 每个 concept 的 definition 是否足够精确？（不能太泛也不能太窄）
2. ☐ 每个 concept 的 principle 是否达到 450-550 字？——这是最重要的评分标准
3. ☐ principle 是否覆盖了完整过程→核心组成→设计原因→适用边界→执行细节这5个维度？
4. ☐ principle 是否纯理论、无类比？（类比在 analogy 中单独写）
5. ☐ 视频涉及几个知识点，就输出了几个 concept？还是合并了/省略了？
6. ☐ 每个类比是否选择了大多数人都熟悉的生活场景？
7. ☐ 类比的局限性是否明确说明？（防止读者过度泛化类比）
8. ☐ "视频完整陈述"中每个 segment 的 content 是否达到 300-500 字？
9. ☐ 思考题是否是开放性的，而不是有标准答案的选择题？
10. ☐ 总字数是否达到约 6000-8000 字？
    - my_analysis.concepts[N].principle: 每个 450-550字 × N个概念
    - video_transcript.segments[N].content: 每个 300-500字 × N个段落
    - 其他字段合计约 500-1000字
```

### 5.2 防止 JSON 解析失败

在 Prompt 中使用结构化的 JSON 格式，并添加：

```
# 重要提醒
- 只输出 JSON，不要有任何额外文字（前言、后记、说明等）
- 如果字段没有内容，用空字符串""或空数组[]，不要省略字段
- 数组元素不要超过5个（concepts不要超过5个，operations不要超过5个）
- 每个content字段至少300字，不要偷工减料
```

### 5.3 提高类比质量

在 Prompt 中添加类比设计指南：

```
# 类比设计指南

类比是帮助理解的关键。一个好的类比必须满足：

1. **场景熟悉度**：目标读者（大多数中国人，有一定技术背景）是否熟悉这个生活场景？
2. **结构相似性**：生活场景中的组成部分，是否能一一对应到原概念的关键部分？
3. **差异明确性**：类比之后，必须立即说明"哪些方面类比无法覆盖"

# 类比示例参考

| 原概念 | 生活类比 | 类比映射 | 局限性 |
|--------|---------|---------|--------|
| 事件循环 | 餐厅唯一厨师 | 厨师=调度器，烤箱=IO等待 | 类比无法覆盖CPU密集计算场景 |
| Docker容器 | 集装箱 | 标准化尺寸=镜像，港口吊机=容器引擎 | 无法覆盖多容器编排 |
| 装饰器 | 相机滤镜 | 滤镜叠加=功能增强，原始照片=原函数 | 无法覆盖有参数的装饰器 |
```

## 七、实施建议

### 7.1 渐进式迁移

考虑到现有数据兼容性，建议分阶段实施：

**阶段1：新增字段兼容**
- 在 `VideoSummary` dataclass 中新增 `my_analysis` 和 `video_transcript` 字段
- 旧字段（summary_cn、key_points_cn 等）保持不变，新字段可为空
- PDF 模板中优先显示新字段，字段为空时降级显示旧字段

**阶段2：Prompt 升级**
- 保持现有 prompt 结构，增加 v2.0 要求的内容
- 新 prompt 同时填充新旧字段
- 观察输出质量，迭代调整

**阶段3：模板升级 + 旧数据重处理**
- PDF 模板重构，优先展示新结构
- 提供脚本批量重处理已处理的视频

### 7.2 兼容策略

为了向后兼容，Prompt 输出应同时包含 v1.0 和 v2.0 的字段：

```json
{
  // v1.0 字段（保持不变）
  "title_cn": "...",
  "summary_cn": "...",
  "key_points_cn": [...],

  // v2.0 新增字段
  "my_analysis": {...},
  "video_transcript": {...}
}
```

PDF 模板中：
- 新字段存在时，渲染 v2.0 模板
- 新字段不存在时，降级渲染 v1.0 模板
